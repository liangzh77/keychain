# 需求规格

## 目标

构建一个带后台服务的网页管理系统，用于集中管理大量大模型 API keys，并按渠道、用户、provider、model 权限向调用系统分发 key。系统需要保留每次分发记录，支持失败上报和历史查询。

## 用户角色

### 管理员

管理员通过后台页面登录系统，可以查看和修改系统数据，包括 provider、model、key、渠道、用户、权限、调用历史和失败记录。

登录账号固定为 `admin`。密码写在 `.env` 文件中，不写入源码仓库。

### 调用系统

调用系统由本项目使用方控制。调用系统负责：

- 提交用户信息。
- 查询用户权限。
- 每次调用大模型前申请一次 key。
- 保证申请到的 key 只用于一次调用。
- 调用大模型失败时上报失败信息。

本系统不代理大模型调用，第一版只负责 key 分发、权限判断、失败记录和审计。

## 核心概念

### Provider

Provider 表示一个大模型服务提供方，例如 OpenAI、Anthropic、Gemini、DeepSeek 等。

每个 provider 可以拥有多个 model 和多个 key。每个 provider 可以独立配置 key 分发策略。

### Model

Model 属于某个 provider。权限粒度使用 `provider + model`。

### Key

Key 属于某个 provider。每个 key 包含：

- 明文 key 值。
- 别名，用于后台和历史记录展示。
- 是否启用。
- 是否可用。
- 排序值。
- 失败次数。
- 最近失败时间。

数据库中允许明文存储 key。后台列表和历史记录不展示完整 key，默认展示星号掩码和 key 别名。

### Channel

Channel 表示一个渠道。每个渠道下可以有多个用户。

渠道可以设置默认权限模式：

- `ALLOW`：默认打开。
- `DENY`：默认关闭。

渠道也可以对具体 provider + model 设置默认权限。

### User

User 属于某个 channel。用户权限可以显式设置，并覆盖渠道默认权限。

### Permission

权限粒度为 `provider + model`。

最终权限判断包含：

1. Provider 是否全局启用。
2. Model 是否全局启用。
3. 用户显式权限。
4. 渠道默认权限。
5. 系统默认关闭。

建议优先级：

```text
用户显式权限 > 渠道 provider+model 默认权限 > 渠道默认权限模式 > 系统默认关闭
```

## Key 分发策略

每个 provider 可配置一种 key 分发策略。

### ROUND_ROBIN

轮询分发。每次申请返回一个可用 key，下次从下一个 key 开始。

适用场景：希望多个 key 均匀消耗。

### STICKY_FIRST_AVAILABLE

优先分发第一个可用 key。只要第一个 key 仍可用，就一直返回它；当它不可用时，才返回下一个可用 key。

适用场景：希望优先使用主 key，其他 key 作为备用。

## 失败上报

调用系统申请 key 后，如果大模型调用失败，需要上报失败。

申请 key 时系统返回 `dispatchLogId`，失败上报应优先使用该 ID 精确关联一次分发记录。

失败上报需要记录：

- 分发记录 ID。
- key。
- provider。
- model。
- 用户。
- 渠道。
- 错误码。
- 错误信息。
- 上报时间。

第一版失败上报先记录失败次数和最近失败时间，由管理员根据失败记录手动恢复或停用 key。后续如果需要自动停用，可以再增加独立配置。

## 历史查询

调用历史需要支持按以下条件过滤：

- 时间范围。
- 用户。
- 渠道。
- Provider。
- Model。
- Key。

历史记录中展示 key 别名，不展示完整明文 key。

## 批量权限管理

后台需要支持批量设置：

- 渠道权限。
- 用户权限。
- Provider 权限。

Provider 权限在数据层仍建议落到 provider + model 粒度。页面可以提供按 provider 批量选择所有 model 的操作。

## 非目标

第一版不做：

- 大模型代理调用。
- Token 用量统计。
- 多管理员账号和角色系统。
- 第三方 OAuth 登录。
- 高可用多节点部署。

这些能力可以在核心分发和审计稳定后扩展。
