# ADR-001：使用 Go、SQLite WAL 和服务端 Key 分发

## 状态

已接受

## 日期

2026-05-06

## 背景

本系统需要管理大量大模型 API keys，并把 key 分发给已有的调用系统。核心需求包括：

- 提供网页后台管理能力。
- 管理多个 providers、models、keys、渠道和用户。
- 按 provider + model 粒度做权限判断。
- 每个 provider 可配置独立的 key 分发策略。
- 记录完整 key 分发历史，并支持调用失败上报。
- 面向学校或内部环境，部署方式需要尽量简单。

调用系统由项目使用方自己控制，因此可以保证每次申请到的 key 只用于一次大模型调用，也可以在 provider 调用失败后主动上报失败信息。

## 决策

后端使用 Go，主数据库使用 SQLite，并开启 WAL。

第一版只做 key 分发，不代理大模型调用。调用系统在每次调用大模型前申请一个 key，并在失败时携带返回的分发记录 ID 上报失败。

后台登录使用单一 `admin` 账号。管理员密码和 session secret 从 `.env` 读取。

API keys 明文存储在数据库中。后台列表和历史页面默认不展示完整 key，只展示 key 别名和星号掩码。

## 备选方案

### PostgreSQL

- 优点：并发能力强，运维模型成熟，后续扩展更容易。
- 缺点：需要单独数据库服务，部署和维护成本更高。
- 结论：MVP 阶段暂不采用，因为当前项目更看重单机部署的简单性。

### 由本系统代理所有大模型调用

- 优点：可以完整控制调用量、失败率、token 统计和限流。
- 缺点：实现范围更大，运行风险更高，还需要适配不同 provider 的请求和响应格式。
- 结论：MVP 阶段暂不采用，因为现有调用系统可控，并能保证一次申请只对应一次调用。

### 多管理员和角色权限系统

- 优点：长期访问控制更细。
- 缺点：会增加产品和安全实现复杂度。
- 结论：MVP 阶段暂不采用。当前场景使用一个 admin 账号即可。

### Key 加密存储

- 优点：数据库文件泄露时更安全。
- 缺点：需要额外设计主密钥管理、备份和恢复流程。
- 结论：MVP 阶段按产品决策暂不采用。Key 明文存入 SQLite，但 UI 默认隐藏完整 key。

## 影响

- SQLite WAL 对 MVP 预期负载应当足够。
- Key 分发和失败上报必须使用数据库事务，避免并发下轮换状态错误。
- 如果后续出现高写入并发或多实例部署，需要重新评估迁移到 PostgreSQL。
- 因为 key 在数据库中明文存储，服务器文件权限、数据库备份和机器访问控制非常重要。
- 调用系统 API 应使用 `.env` 中的内部 token 保护，并与后台 admin session 登录分开。
