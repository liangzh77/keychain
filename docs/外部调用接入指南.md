# Keychain 外部调用接入指南

本文档面向“调用系统”或其他外部业务系统。外部系统只应调用 `/api/runtime/*` 接口，不应调用后台管理接口。

## 基本信息

生产地址：

```text
https://keychain.liangz77.cn
```

所有 Runtime API 都返回 JSON，并要求携带内部共享 token：

```http
Authorization: Bearer <RUNTIME_API_TOKEN>
Content-Type: application/json
```

`RUNTIME_API_TOKEN` 由 Keychain 服务端 `.env` 配置。不要把 token 写进代码仓库。

## 错误格式

错误响应统一为：

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "channel id is required"
  }
}
```

常见状态码：

- `200`：成功。
- `400`：JSON 格式错误。
- `401`：缺少或错误的 Runtime token。
- `403`：用户没有调用该 provider/model 的权限。
- `404`：资源不存在。
- `422`：请求语义不合法，例如用户被停用、没有可用 key。
- `500`：服务端错误。

## 1. 提交或更新用户

外部系统在第一次使用某个用户前，应先提交用户信息。重复提交同一个 `channelId + externalUserId` 会更新用户名称和启用状态。

```http
POST /api/runtime/users
```

请求：

```json
{
  "channelId": "channel_001",
  "externalUserId": "student-001",
  "name": "学生 001"
}
```

字段说明：

- `channelId`：渠道 ID，必填。
- `externalUserId`：外部系统自己的用户 ID。可选；不传时默认等于 `name`。
- `name`：用户名称，必填。
- `isEnabled`：是否启用。可选，默认 `true`。

响应：

```json
{
  "id": "user_001",
  "channelId": "channel_001",
  "externalUserId": "student-001",
  "name": "学生 001",
  "isEnabled": true
}
```

后续申请 key 时使用响应里的 `id` 作为 `userId`。

## 2. 查询用户权限

```http
GET /api/runtime/users/{userId}/permissions
```

响应：

```json
{
  "userId": "user_001",
  "permissions": [
    {
      "providerId": "provider_001",
      "providerName": "OpenAI",
      "modelId": "model_001",
      "modelName": "gpt-4.1",
      "allowed": true
    }
  ]
}
```

权限计算规则：

1. 用户显式权限优先。
2. 如果没有用户显式权限，使用渠道对该 provider/model 的默认权限。
3. 如果渠道也没有单独配置该 provider/model，使用渠道的默认权限模式。

## 3. 查询可用 Providers

```http
GET /api/runtime/providers
```

响应：

```json
[
  {
    "id": "provider_001",
    "name": "OpenAI"
  }
]
```

只返回已启用的 providers。

## 4. 查询可用 Models

```http
GET /api/runtime/models?providerId=provider_001
```

响应：

```json
[
  {
    "id": "model_001",
    "providerId": "provider_001",
    "name": "gpt-4.1"
  }
]
```

`providerId` 必填。接口只返回指定 provider 下已启用的 models。

## 5. 申请一次调用 Key

外部系统每次调用大模型前，都应先向 Keychain 申请一次 key。调用系统需要保证申请到的 key 只用于这一次模型调用。

```http
POST /api/runtime/dispatch-key
```

请求：

```json
{
  "channelId": "channel_001",
  "userId": "user_001",
  "providerId": "provider_001",
  "modelId": "model_001"
}
```

响应：

```json
{
  "dispatchLogId": "dispatch_001",
  "providerName": "OpenAI",
  "modelName": "gpt-4.1",
  "keyId": "key_001",
  "keyAlias": "openai-main-01",
  "key": "sk-real-key"
}
```

说明：

- `key` 是明文密钥，只在 Runtime API 中返回。
- `dispatchLogId` 必须保存。模型调用失败时，要用它上报失败。
- Keychain 会根据 provider 的分发策略选择 key：
  - `ROUND_ROBIN`：可用 key 轮询分发。
  - `STICKY_FIRST_AVAILABLE`：一直使用第一个可用 key，直到它不可用。

## 6. 上报调用失败

如果外部系统使用申请到的 key 调用大模型失败，应立即上报。

```http
POST /api/runtime/key-failures
```

请求：

```json
{
  "dispatchLogId": "dispatch_001",
  "errorCode": "rate_limit",
  "errorMessage": "provider returned 429"
}
```

响应：

```json
{
  "reported": true,
  "keyId": "key_001",
  "keyAlias": "openai-main-01",
  "isAvailable": false
}
```

当前实现中，失败上报会把本次 dispatch 使用的 key 标记为不可用，并记录失败历史。

## 推荐调用流程

1. 调用 `POST /api/runtime/users` 创建或更新外部用户，拿到 `userId`。
2. 可选：调用 `GET /api/runtime/users/{userId}/permissions` 展示用户权限。
3. 调用 `GET /api/runtime/providers` 和 `GET /api/runtime/models?providerId=...` 获取可用模型。
4. 每次模型调用前，调用 `POST /api/runtime/dispatch-key` 申请 key。
5. 使用返回的 `key` 调用大模型。
6. 如果大模型调用失败，调用 `POST /api/runtime/key-failures` 上报。

## curl 示例

```bash
curl -sS https://keychain.liangz77.cn/api/runtime/providers \
  -H "Authorization: Bearer $RUNTIME_API_TOKEN"
```

```bash
curl -sS https://keychain.liangz77.cn/api/runtime/users \
  -H "Authorization: Bearer $RUNTIME_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"channelId":"channel_001","externalUserId":"student-001","name":"学生 001"}'
```

```bash
curl -sS https://keychain.liangz77.cn/api/runtime/dispatch-key \
  -H "Authorization: Bearer $RUNTIME_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"channelId":"channel_001","userId":"user_001","providerId":"provider_001","modelId":"model_001"}'
```
