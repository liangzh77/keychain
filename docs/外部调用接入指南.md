# Keychain 外部调用接入指南

本文档面向“调用系统”或其他外部业务系统。外部系统只应调用 `/api/runtime/*` 接口，不应调用后台管理接口。

## 基本信息

生产地址：

```text
https://keychain.liangz77.cn
```

所有 Runtime API 都返回 JSON，并要求携带内部共享 token：

```http
Authorization: Bearer <RUNTIME_API_TOKEN>
Content-Type: application/json
```

`RUNTIME_API_TOKEN` 对应 Keychain 服务端配置项 `KEYCHAIN_RUNTIME_API_TOKEN`。这是生产服务的调用密钥，应由 Keychain 管理员通过私密渠道提供给外部系统负责人。

外部系统应把它配置在自己的服务端环境变量中，例如：

```bash
KEYCHAIN_RUNTIME_API_TOKEN=<由 Keychain 管理员提供的真实 token>
```

然后在服务端请求 Runtime API 时带入：

```bash
Authorization: Bearer $KEYCHAIN_RUNTIME_API_TOKEN
```

不要把真实 token 写进代码仓库、前端代码、公开文档、接口文档样例或浏览器环境。本文档只说明变量名和使用方式，不保存生产 token 明文。

## 错误格式

错误响应统一为：

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "channel id is required"
  }
}
```

常见状态码：

- `200`：成功。
- `400`：JSON 格式错误。
- `401`：缺少或错误的 Runtime token；托管用户登录时用户名或密码错误。
- `403`：用户没有调用该 provider/model 的权限。
- `404`：资源不存在。
- `409`：资源冲突，例如同一渠道中托管用户名已存在。
- `422`：请求语义不合法，例如用户被停用、没有可用 key。
- `500`：服务端错误。

## 用户系统模式

渠道有两种用户系统模式：

- `EXTERNAL_MANAGED`：外部系统自有用户系统。外部系统负责注册、登录、更新、注销用户，并把用户信息同步到 Keychain。
- `KEYCHAIN_HOSTED`：Keychain 托管用户系统。外部系统调用 Keychain 的注册、登录、重置密码、注销接口。

两种模式创建出来的用户在 Keychain 内部都是同一种用户，都使用响应中的 `id` 作为后续接口的 `userId`，权限管理、key 授权、申请 key 的流程完全一致。

## 1. 提交或更新外部用户

外部系统维护用户主数据。新增用户或修改用户信息、启用状态时，调用这个接口提交单个用户信息。若同一渠道内已经存在相同 `externalUserId` 的用户，则更新；否则创建。

该接口只适用于渠道的用户系统模式为 `EXTERNAL_MANAGED` 的场景。

```http
PUT /api/runtime/channels/{channelId}/external-users/{externalUserId}
```

请求：

```json
{
  "name": "学生 001",
  "isEnabled": true
}
```

字段说明：

- `channelId`：渠道 ID，放在 URL 中，必填。
- `externalUserId`：外部系统中的用户 ID，放在 URL 中，必填。同一渠道内必须唯一。
- `name`：用户名称，可选。不传时使用 `externalUserId`。
- `isEnabled`：是否启用。可选，默认 `true`。传 `false` 表示停用该用户。

响应：

```json
{
  "id": "user_001",
  "channelId": "channel_001",
  "externalUserId": "student_001",
  "name": "学生 001",
  "isEnabled": true
}
```

后续申请 key 时使用响应中的 `id` 作为 `userId`。

## 2. 删除外部用户

外部系统中的用户注销或删除后，调用这个接口删除 Keychain 中对应用户。

该接口只适用于渠道的用户系统模式为 `EXTERNAL_MANAGED` 的场景。

```http
DELETE /api/runtime/channels/{channelId}/external-users/{externalUserId}
```

响应：

```json
{
  "deleted": true
}
```

## 3. 注册托管用户

该接口只适用于渠道的用户系统模式为 `KEYCHAIN_HOSTED` 的场景。

```http
POST /api/runtime/channels/{channelId}/hosted-users/register
```

请求：

```json
{
  "username": "student_001",
  "name": "学生 001",
  "password": "user-password"
}
```

字段说明：

- `channelId`：渠道 ID，放在 URL 中，必填。
- `username`：登录用户名，必填。同一渠道内必须唯一。
- `name`：用户名称，可选。不传时使用 `username`。
- `password`：登录密码，必填。Keychain 只保存密码哈希，不保存明文密码。

响应：

```json
{
  "id": "user_001",
  "channelId": "channel_001",
  "externalUserId": "student_001",
  "name": "学生 001",
  "isEnabled": true
}
```

说明：托管用户的 `externalUserId` 字段等于 `username`。后续申请 key 时使用响应中的 `id` 作为 `userId`。

## 4. 登录托管用户

该接口用于校验托管用户的用户名和密码。登录成功后返回 Keychain 用户信息；Keychain 不下发终端用户 session token，外部系统是否建立自己的登录态由外部系统决定。

```http
POST /api/runtime/channels/{channelId}/hosted-users/login
```

请求：

```json
{
  "username": "student_001",
  "password": "user-password"
}
```

响应：

```json
{
  "id": "user_001",
  "channelId": "channel_001",
  "externalUserId": "student_001",
  "name": "学生 001",
  "isEnabled": true
}
```

用户名或密码错误时返回 `401`。

## 5. 重置托管用户密码

```http
POST /api/runtime/channels/{channelId}/hosted-users/{userId}/reset-password
```

请求：

```json
{
  "password": "new-user-password"
}
```

响应：

```json
{
  "id": "user_001",
  "channelId": "channel_001",
  "externalUserId": "student_001",
  "name": "学生 001",
  "isEnabled": true
}
```

## 6. 注销托管用户

```http
DELETE /api/runtime/channels/{channelId}/hosted-users/{userId}
```

响应：

```json
{
  "deleted": true
}
```

## 7. 查询用户权限

```http
GET /api/runtime/users/{userId}/permissions
```

响应：

```json
{
  "userId": "user_001",
  "permissions": [
    {
      "providerId": "provider_001",
      "providerName": "OpenAI",
      "modelId": "model_001",
      "modelName": "gpt-4.1",
      "allowed": true
    }
  ]
}
```

权限计算规则：

1. 用户显式权限优先。
2. 如果没有用户显式权限，使用渠道对该 provider/model 的默认权限。
3. 如果渠道也没有单独配置该 provider/model，使用渠道的默认权限模式。

## 8. 查询可用 Providers

```http
GET /api/runtime/providers
```

响应：

```json
[
  {
    "id": "provider_001",
    "name": "OpenAI"
  }
]
```

只返回已启用的 providers。

## 9. 查询可用 Models

```http
GET /api/runtime/models?providerId=provider_001
```

响应：

```json
[
  {
    "id": "model_001",
    "providerId": "provider_001",
    "name": "gpt-4.1"
  }
]
```

`providerId` 必填。接口只返回指定 provider 下已启用的 models。

## 10. 申请一次调用 Key

外部系统每次调用大模型前，都应先向 Keychain 申请一次 key。调用系统需要保证申请到的 key 只用于这一次模型调用。

```http
POST /api/runtime/dispatches
```

请求：

```json
{
  "channelId": "channel_001",
  "userId": "user_001",
  "providerId": "provider_001",
  "modelId": "model_001"
}
```

响应：

```json
{
  "dispatchLogId": "dispatch_001",
  "providerName": "OpenAI",
  "modelName": "gpt-4.1",
  "keyId": "key_001",
  "keyAlias": "openai-main-01",
  "key": "sk-real-key"
}
```

说明：

- `key` 是明文密钥，只在 Runtime API 中返回。
- `dispatchLogId` 必须保存。模型调用失败时，要用它上报失败。
- Keychain 会根据 provider 的分发策略选择 key：
  - `ROUND_ROBIN`：可用 key 轮询分发。
  - `STICKY_FIRST_AVAILABLE`：一直使用第一个可用 key，直到它不可用。
- 如果后台为该用户配置了 provider 下的 key 授权，Keychain 只会从被允许的 key 中分发。没有显式 key 授权记录时，默认允许该 provider 下所有可用 key。

## 11. 上报调用失败

如果外部系统使用申请到的 key 调用大模型失败，应立即上报。

```http
POST /api/runtime/dispatches/{dispatchLogId}/failure
```

请求：

```json
{
  "errorCode": "rate_limit",
  "errorMessage": "provider returned 429"
}
```

响应：

```json
{
  "reported": true,
  "keyId": "key_001",
  "keyAlias": "openai-main-01",
  "isAvailable": false
}
```

当前实现中，失败上报会把本次 dispatch 使用的 key 标记为不可用，并记录失败历史。

## 推荐调用流程

外部自有用户系统：

1. 调用 `PUT /api/runtime/channels/{channelId}/external-users/{externalUserId}` 新增或更新单个外部用户，拿到 `userId`。
2. 用户删除时，调用 `DELETE /api/runtime/channels/{channelId}/external-users/{externalUserId}`。
3. 可选：调用 `GET /api/runtime/users/{userId}/permissions` 展示用户权限。
4. 调用 `GET /api/runtime/providers` 和 `GET /api/runtime/models?providerId=...` 获取可用模型。
5. 每次模型调用前，调用 `POST /api/runtime/dispatches` 申请 key。
6. 使用返回的 `key` 调用大模型。
7. 如果大模型调用失败，调用 `POST /api/runtime/dispatches/{dispatchLogId}/failure` 上报。

Keychain 托管用户系统：

1. 用户注册时，调用 `POST /api/runtime/channels/{channelId}/hosted-users/register`，拿到 `userId`。
2. 用户登录时，调用 `POST /api/runtime/channels/{channelId}/hosted-users/login` 校验用户名和密码，拿到 `userId`。
3. 用户忘记密码或管理员重置密码时，调用 `POST /api/runtime/channels/{channelId}/hosted-users/{userId}/reset-password`。
4. 用户注销时，调用 `DELETE /api/runtime/channels/{channelId}/hosted-users/{userId}`。
5. 后续查询权限、查询 provider/model、申请 key、失败上报流程与外部自有用户系统一致。

## 安全建议

- 所有 Runtime API 都应只通过 HTTPS 调用。
- `RUNTIME_API_TOKEN` 是服务间共享密钥，应保存在服务端配置或密钥管理系统中，不要放进前端代码。
- 托管用户接口的 `password` 不要写入业务日志、访问日志或错误日志。
- 托管用户登录接口只校验凭据并返回用户信息；如果外部系统需要浏览器登录态，应由外部系统自己签发和维护 session。

## curl 示例

```bash
curl -sS https://keychain.liangz77.cn/api/runtime/providers \
  -H "Authorization: Bearer $RUNTIME_API_TOKEN"
```

```bash
curl -sS -X PUT https://keychain.liangz77.cn/api/runtime/channels/channel_001/external-users/student_001 \
  -H "Authorization: Bearer $RUNTIME_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"学生 001","isEnabled":true}'
```

```bash
curl -sS -X POST https://keychain.liangz77.cn/api/runtime/channels/channel_001/hosted-users/register \
  -H "Authorization: Bearer $RUNTIME_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"username":"student_001","name":"学生 001","password":"user-password"}'
```

```bash
curl -sS -X POST https://keychain.liangz77.cn/api/runtime/channels/channel_001/hosted-users/login \
  -H "Authorization: Bearer $RUNTIME_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"username":"student_001","password":"user-password"}'
```

```bash
curl -sS -X POST https://keychain.liangz77.cn/api/runtime/dispatches \
  -H "Authorization: Bearer $RUNTIME_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"channelId":"channel_001","userId":"user_001","providerId":"provider_001","modelId":"model_001"}'
```
