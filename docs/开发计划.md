# 开发计划

## 概览

本计划把 Keychain 拆成可逐步提交、可验证的开发阶段。优先顺序是先打牢 Go 服务、配置、SQLite WAL、迁移和认证，再实现管理 API、运行时 key 分发 API，最后补后台页面、测试和发布准备。

## 总体原则

- 每个阶段完成后都应保持项目可构建、可运行。
- 关键业务逻辑优先写测试，尤其是权限判断、key 轮换和失败上报。
- 数据库迁移、API 契约和页面实现保持同步。
- 后台管理 API 使用 admin session 保护，运行时 API 使用 `RUNTIME_API_TOKEN` 保护。
- Key 分发和失败上报必须使用数据库事务。

## 阶段 1：项目骨架

**目标：** 建立最小可运行 Go 服务，为后续 API 和页面提供基础。

**任务：**

- 初始化 Go module。
- 建立目录结构，例如 `cmd/server`、`internal/config`、`internal/http`、`internal/db`。
- 实现 `.env` 配置加载，读取 `ADMIN_USERNAME`、`ADMIN_PASSWORD`、`SESSION_SECRET`、`RUNTIME_API_TOKEN`、`DATABASE_PATH`。
- 实现基础 HTTP server。
- 增加结构化日志和统一错误响应工具。
- 实现 `GET /api/health`。

**验收标准：**

- `go test ./...` 通过。
- 服务可启动。
- `GET /api/health` 返回 `status=ok`。
- 缺少必要环境变量时启动失败并给出清晰错误。

**建议提交：**

```text
feat: initialize go service skeleton
```

## 阶段 2：SQLite 和迁移

**目标：** 建立 SQLite WAL 数据层和可重复执行的数据库初始化流程。

**任务：**

- 建立 SQLite 连接初始化。
- 启动时执行 `PRAGMA journal_mode = WAL`、`PRAGMA busy_timeout`、`PRAGMA foreign_keys = ON`。
- 实现迁移机制。
- 按 `docs/数据库设计.md` 创建核心表和索引。
- 增加数据库健康检查，接入 `/api/health` 的 `database` 字段。

**验收标准：**

- 空数据库启动后自动完成迁移。
- 重复启动不会重复创建或破坏表结构。
- 外键约束生效。
- `/api/health` 能检查数据库连通性。

**建议提交：**

```text
feat: add sqlite wal migrations
```

## 阶段 3：admin 登录和鉴权

**目标：** 后台页面和管理 API 必须登录后才能访问。

**任务：**

- 实现 `POST /api/auth/login`。
- 实现 `POST /api/auth/logout`。
- 实现 `GET /api/auth/me`。
- 实现 session token 生成、hash 存储和过期清理。
- 设置 httpOnly、SameSite cookie。
- 增加后台管理 API 鉴权中间件。
- 增加登录失败的基础限流或延迟策略。

**验收标准：**

- 正确 admin 密码可以登录。
- 错误密码无法登录。
- 未登录访问管理 API 返回 `401`。
- 登录后访问管理 API 放行。
- logout 后 session 失效。

**建议提交：**

```text
feat: add admin session authentication
```

## 阶段 4：Provider、Model、Key 管理 API

**目标：** 管理员可以维护 provider、model 和 key。

**任务：**

- 实现 provider CRUD。
- 实现 model CRUD。
- `GET /api/models` 必须要求 `providerId` 或 `providerCode`。
- 实现 key CRUD。
- key 列表返回 `maskedValue`，不返回完整 `secretValue`。
- 支持 key 的 `alias`、`isEnabled`、`isAvailable`、`sortOrder`。
- provider 支持 `rotationStrategy`。

**验收标准：**

- provider code 唯一。
- model code 在同一 provider 下唯一。
- key alias 在同一 provider 下唯一。
- model 列表不传 provider 过滤时返回 `422`。
- key 列表不会泄露完整明文 key。

**建议提交：**

```text
feat: add provider model key admin APIs
```

## 阶段 5：Channel、User、Permission 管理 API

**目标：** 管理员可以维护渠道、用户和 provider + model 粒度权限。

**任务：**

- 实现 channel CRUD。
- 实现 user CRUD。
- 实现 user permissions CRUD。
- 实现 channel permission defaults CRUD。
- 实现批量渠道权限接口。
- 实现批量用户权限接口。
- 实现批量 provider 权限接口，后端展开为 provider 下所有 model。
- 实现权限判断服务，统一给后台和运行时 API 使用。

**验收标准：**

- 用户必须属于一个渠道。
- `defaultPermissionMode` 支持 `ALLOW` 和 `DENY`。
- 权限优先级符合文档：用户显式权限 > 渠道 provider+model 默认权限 > 渠道默认权限模式 > 默认拒绝。
- 批量权限操作具备事务性，要么全部成功，要么全部失败。

**建议提交：**

```text
feat: add channel user permission APIs
```

## 阶段 6：运行时 API

**目标：** 调用系统可以提交用户、查询权限、查询 provider/model、申请 key、上报失败。

**任务：**

- 实现运行时 API token 鉴权中间件。
- 实现 `POST /api/runtime/users`。
- 实现 `GET /api/runtime/users/:id/permissions`。
- 实现 `GET /api/runtime/providers`。
- 实现 `GET /api/runtime/models`，必须按 provider 过滤。
- 实现 `POST /api/runtime/dispatch-key`。
- 实现 `POST /api/runtime/key-failures`。

**验收标准：**

- 未带正确 `RUNTIME_API_TOKEN` 时返回 `401`。
- 无权限申请 key 时返回 `403`。
- provider/model/key 不可用时不会分发 key。
- 申请 key 成功后生成 dispatch log。
- 失败上报可通过 `dispatchLogId` 精确关联一次分发。

**建议提交：**

```text
feat: add runtime key dispatch APIs
```

## 阶段 7：Key 分发和失败上报测试

**目标：** 验证系统最关键的事务逻辑。

**任务：**

- 测试 `ROUND_ROBIN`：多个可用 key 按顺序轮换。
- 测试 `STICKY_FIRST_AVAILABLE`：一直使用排序最前的可用 key。
- 测试禁用 key、不可用 key 不参与分发。
- 测试并发申请不会破坏轮换 cursor。
- 测试失败上报会写入 failure report，并更新 key 失败统计。
- 测试重复或无效 `dispatchLogId` 的错误处理。

**验收标准：**

- 关键分发测试稳定通过。
- 并发测试能覆盖至少几十个并发申请。
- 失败上报不会泄露完整 key。

**建议提交：**

```text
test: cover key dispatch and failure reporting
```

## 阶段 8：后台页面

**目标：** 提供可用的网页后台，完成日常管理工作流。

**任务：**

- 登录页。
- 总览页，展示 provider、key、用户、调用记录概要。
- Provider/model 管理页。
- Key 管理页，默认星号展示 key，支持新增、编辑、启用、停用、删除。
- Channel/user 管理页。
- 权限管理页，支持用户权限、渠道默认权限和批量设置。
- 调用历史页，支持时间、用户、渠道、provider、model、key 过滤。
- 失败记录页，支持查看错误码、错误信息和 key 别名。

**验收标准：**

- 未登录访问后台页面会跳转登录页。
- 页面不会展示完整 key，除非在创建或替换 key 的表单中由管理员主动输入。
- 批量权限设置有确认反馈和错误提示。
- 历史查询过滤条件与 API 保持一致。

**建议提交：**

```text
feat: add admin web console
```

## 阶段 9：接口文档、种子数据和运维准备

**目标：** 让项目可以被部署和交接。

**任务：**

- 更新 README 的启动命令、配置说明和常见问题。
- 增加可选种子数据脚本或命令。
- 增加生产部署说明。
- 增加数据库备份说明。
- 明确 `.env` 配置项含义。
- 增加基础 release checklist。

**验收标准：**

- 新环境可以按 README 启动服务。
- `.env.example` 覆盖所有必要配置。
- 文档说明 SQLite 数据库文件、WAL 文件和备份注意事项。

**建议提交：**

```text
docs: add deployment and operations guide
```

## 检查点

### 基础检查点：阶段 1-3 后

- `go test ./...` 通过。
- 服务可启动。
- `/api/health` 正常。
- admin 登录、登出、鉴权正常。

### 核心 API 检查点：阶段 4-6 后

- 管理 API 可维护全部核心数据。
- 运行时 API 可完成申请 key 和失败上报。
- 权限判断符合规格。
- 不存在完整 key 从列表或历史接口泄露。

### 发布前检查点：阶段 7-9 后

- 关键事务测试通过。
- 后台主要流程可手动跑通。
- README 和 `.env.example` 与实现一致。
- 数据库文件、`.env`、构建产物不会被提交。

## 风险和缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| SQLite 并发写入锁冲突 | key 分发失败或延迟 | 开启 WAL、设置 busy timeout、缩短事务时间、关键路径加并发测试 |
| key 轮换状态错误 | 同一个策略下分发不符合预期 | cursor 读取和更新放在同一事务内，覆盖轮询和并发测试 |
| 权限优先级误解 | 用户拿到不该拿的 provider/model | 权限判断集中在一个服务，测试覆盖用户显式权限和渠道默认权限 |
| key 明文存储 | 数据库泄露时风险高 | UI 默认掩码、`.gitignore` 排除数据库、部署文档强调文件权限和备份保护 |
| 后台页面与 API 契约漂移 | 页面操作失败或数据错误 | 先稳定 API 契约，再实现页面，页面使用同一类型定义或客户端封装 |

## 暂不实现

- 多管理员账号和角色系统。
- 代理大模型调用。
- token 用量统计。
- 自动根据失败阈值停用 key。
- 多实例部署和 PostgreSQL 迁移。

这些内容可以在 MVP 稳定后作为后续迭代。
